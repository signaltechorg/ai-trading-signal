### 1 — Pine `ta.stoch()` wrong arity (HIGH)
`ta.stoch(close, high, low, stochKLen, stochDLen, stochSmooth)` passes 6 args;
Pine v5 `ta.stoch` takes exactly 4 `(source, high, low, length)`. Pine is strict on
built-in arity — this is a hard compile error that aborts the *entire* generated
indicator, so TC-233 export is broken for every user (copy + `.pine` download).
The finder's suggested reorder was itself wrong (it moved `high` to source); the
verifier confirmed the existing first four args are already correctly ordered, so
the fix is to delete the two trailing args → `ta.stoch(close, high, low, stochKLen)`.

### 2 — Pine `ta.bb()` tuple unpacked in wrong order (MEDIUM)
`[bbUpper, bbMid, bbLower] = ta.bb(...)` but `ta.bb` returns `[middle, upper, lower]`.
So `bbUpper` holds the midline; the SELL check `close >= bbUpper` fires against the
midline (true ~half the time) instead of the upper band, inflating sellScore in the
exported script. Fix: `[bbMid, bbUpper, bbLower] = ta.bb(close, bbLen, bbMult)`.

### 3 — TradingView auto-follow not idempotent on replay (MEDIUM)
`premium_signals` dedups via `ON CONFLICT (source_id) DO NOTHING`, but `autoFollowSignal()`
runs unconditionally afterward with no check whether the row was new. `openPosition`
inserts a fresh random-id paper position with no `signal_id` dedup, so a redelivered
webhook opens a duplicate demo position, skewing the public widget's win-rate/equity.
Only affects the demo user (`PUBLIC_WIDGET_DEMO_USER_ID`); dormant if unset. Fix:
`INSERT ... RETURNING source_id` via `queryOne`; only auto-follow when a row was inserted.

### 4 — Auto-follow discards signal entry price (MEDIUM, data-integrity)
`autoFollowSignal` receives `entry: number` but the `openPosition` call omits
`entryPrice`, so the position opens at `BASE_PRICES[symbol] ?? 100`. The verifier
disproved the finder's "immediate false SL/TP closure" claim (closure logic reads
only SL/TP/currentPrice, not entry), but confirmed the real harm: `closePosition`
computes `movePct` off the wrong entry, corrupting recorded PnL%/equity for the demo
copy-trading preview and the manual follow-signal endpoint. `autoFollowSignal` itself
predates the window, but the in-window TV-webhook wiring makes it reachable in prod.
Fix: forward `entryPrice: signal.entry` (one line).

### 5 — Bearer admin auth dead-for-purpose (MEDIUM, broken contract)
`middleware.ts:232` verifies Bearer tokens with `verifyAdminSession`, which requires
the signed `admin.<ts>.<sig>` 3-part format. A raw `ADMIN_SECRET` Bearer (no dots)
always fails, contradicting the inline comment "still compared against the raw secret
for API clients". Regression from 50d97d30; login only ever sets the signed token in
the httpOnly cookie, so no client can obtain a Bearer-usable token. Fails CLOSED (no
bypass) — denies documented programmatic/curl admin access. Fix: try signed verify,
then fall back to a constant-time raw-secret compare (Edge-safe; `node:crypto` is
unavailable in middleware). Routed through security-reviewer.

### 6 — Referral link hardcodes `tradeclaw.win` (MEDIUM, inconsistency)
`referrals/route.ts:43` builds `https://tradeclaw.win/pricing?ref=...` while every
other functional URL uses `process.env.NEXT_PUBLIC_BASE_URL ?? 'https://tradeclaw.win'`.
Self-hosted/multi-domain deployments emit referral links pointing at upstream.
No impact on hosted prod. Fix: use the env-var pattern.

### 7 — Discord broadcast send-then-mark can duplicate (MEDIUM)
`discord-broadcast.ts` sends then `UPDATE ... discord_posted_at = NOW()`. The cron
double-fires inside its 4h slot (~:00 and ~:05 via the 5-min sync timer); if the send
succeeds but the UPDATE throws, the row stays `NULL` and the next tick re-sends —
violating the file's own "at most once per 2h window" contract. The finder's primary
fix (try/catch + continue) does NOT prevent the duplicate. Fix: atomic claim —
`UPDATE ... SET discord_posted_at = NOW() WHERE id = $1 AND discord_posted_at IS NULL
RETURNING id`; send only if claimed; release the claim on send failure so it retries.

### 8 — precompute caches under a profileId it didn't generate with (LOW, REPORT)
`signal-worker.ts:31` computes `profileId = safeProfileId(preset.id)` but line 36 calls
`getSignals({}, …)` without it, then caches the payload tagged with that profileId.
Fully dormant today: `safeProfileId` collapses every preset to `classic` and
`getSignals` defaults to `classic`, so the tag always matches. Will become a permanent
cache miss + mistagged entry once a real second profile is added. On the load-bearing
signal-cache path and not currently broken, so reported, not auto-fixed.
One-line fix when a second profile lands: `getSignals({ profileId }, { skipCache: true })`.

### 9 — Stale `confluenceBonus` comment (LOW, FIX)
`signal-generator.ts:40` comment lists `+15, +5, 0, -20` but the 4-TF logic also
assigns `+10` (3/4 agreement). Comment-only; update to `+15, +10, +5, 0, -20`.

### 10 — Stale `DelayCountdown` JSDoc (LOW, REPORT)
`DelayCountdown.tsx:25` JSDoc says "15-min delay"; the free tier is 30 min
(`tier.ts:56`, caller passes `30 * 60 * 1000`). Runtime-correct (uses `delayMs` prop).
File was not touched in the 2-week window — logged as a follow-up rather than a
drive-by edit. One-word fix: "15-min" → "30-min" (or "free-tier delay").

### 11 — Referral total includes cancelled revenue (LOW, FIX)
`db.ts:875` sums `totalShareCents` over all rows while pending/paidOut filter by status,
so a `cancelled` row would inflate "Total Earnings" and break Total = Pending + Paid Out.
Latent (no code path sets status to non-pending yet — insert-only), but a one-line
defensive fix prevents a future financial-display bug: exclude `cancelled` from the total.
